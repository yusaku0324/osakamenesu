# 大阪メンエス.com 機能バックログ（優先度付き）

本書は要件定義で挙げた未対応項目を整理し、優先度と初手タスクをまとめたものです。優先度は P0 (最優先) / P1 / P2 / P3 の 4 段階で定義します。

| ID | 項目 | 優先度 | 背景/期待効果 | 初手タスク | 所感 |
| --- | --- | --- | --- | --- | --- |
| BL-001 | 予約通知連携（メール/SMS/LINE） | **P0** | 予約フォーム送信後に店舗側が即時把握できず、機会損失リスクが高い。 | - 通知チャネル候補を決定（メール + LINE Notify など）<br>- 通知テンプレートと送信先を設定可能にする DB 項目を追加<br>- FastAPI 側で予約作成時に非同期送信処理を実装 | 通知遅延は UX 直結。早期に最小チャネル（メール）だけでも稼働させる。 |
| BL-002 | 店舗セルフ管理 UI（ダッシュボード） | **P0** | 現状は運営が API/key で手動更新しており、店舗回転が上がると運用負荷が破綻する。 | - 店舗アカウント概念と認証方式を決定（メール + Magic Link など）<br>- Next.js 側に `/dashboard` を用意しプロフィール・シフト・日記編集を行えるフォームを設計<br>- 権限チェック付き API エンドポイントを追加 | 大型機能だが今後のスケールに必須。段階的リリースを想定。 |
| BL-003 | エンドユーザ会員ログイン/お気に入り | **P1** | 会員機能が無いため、再訪や予約履歴の管理ができない。顧客ロイヤリティを高めたい。 | - 利用者アカウントモデルを設計（メール or LINE 連携）<br>- お気に入り店舗/閲覧履歴の保存スキーマ策定<br>- フロントで「お気に入り」「閲覧履歴」UI を追加 | 店舗側成果指標を高めるための中期改善。セルフ管理 UI の後着手。 |
| BL-004 | 地図・駅別検索 / 距離ソート | **P1** | エリア粒度が粗く、駅近や現在地周辺検索ができない。競合（シティヘブン等）との差別化に必要。 | - `GeoLocation` を元に Meilisearch で距離ソートできるようスキーマ更新<br>- Map コンポーネント (Mapbox/Google Maps) の導入検証<br>- 駅コード辞書を整備して facet/フィルタに追加 | 位置情報を既に保持しているため実装可能。UI 作り込みに時間を要する。 |
| BL-005 | 広告メニュー管理（PR枠/クーポン管理） | **P2** | 現状プロモーションは手動 JSON 管理。広告課金を見据えた枠管理・レポーティングが無い。 | - PR 枠の種別と表示ロジックを要件化<br>- 課金プラン/露出期間を管理するテーブル追加<br>- 表示ログ計測とダッシュボード案を作成 | 収益化ポイント。セルフ管理 UI と連動させると効果大。 |
| BL-006 | 観測性強化（エラーレポート・トラッキング） | **P2** | GA のみだと API 障害や予約失敗が検知できない。品質リスク。 | - Sentry 等の APM 導入検討<br>- API ログの構造化（JSON logging）<br>- 予約エラー時の告知チャネル整備 | 優先度は中程度だが早めに仕込みたい。通知基盤とセットで進める。 |
| BL-007 | コンテンツ SEO / CMS 連携 | **P3** | 初めての方向け記事やブログを手作業で更新予定。CMS 化で更新効率を上げたい。 | - CMS 選定（Contentful / microCMS / Notion API 等）<br>- `/demo/blog` の静的生成/ISR 設計<br>- 記事コンポーネントのデザイン整備 | 他優先項目の後。MVP 成功後に強化。 |

### ロードマップ実行の補足

- 各フェーズ開始前に前フェーズのリグレッション／モニタリングを 1~2 日確保し、品質確保と Doc 更新を必ず実施。
- Phase 1 と 2 は密接に連動するため、通知設定値のスキーマ（DB/環境変数）を先に揃えておく。
- Phase 3 の会員基盤は将来のポイント/レビュー機能にも直結するため、拡張性を意識してスキーマ設計を行う。
- Phase 4 では駅データセット・地図 API 利用料など外部コストが発生する可能性があるため、着手前に概算見積もりをまとめて経営判断を仰ぐ。

### Phase 1 (BL-001) 詳細タスク

1. **要件整理 & チャネル決定（Day 1-3）**
   - 送信対象（店舗担当・運営）と通知メッセージ項目を定義。
   - メール送信サービス（SMTP / SendGrid / AWS SES 等）を比較し採用案を決定。暫定は SMTP でスタートしつつ、API ベースの代替も評価。
   - LINE Notify / Slack Webhook の可否確認（トークン管理、利用規約、費用）。

2. **設定スキーマとシークレット整備（Day 2-4）**
   - `.env` に送信元メール・SMTP 認証・LINE トークンなどを追加、`docs/requirements.md` に反映。
   - 店舗ごとの通知先設定を保存するテーブル/カラム設計（例: `shop_notification_settings`）。ダッシュボード連携を見据え API で CRUD 可能な形にする。

3. **メールテンプレート設計（Day 4-5）**
   - 予約リクエスト通知テンプレート（件名/本文/サマリ表）を Markdown or Jinja2 で作成。
   - 多言語化要否を確認。最低限、事務局向けと店舗向けで表示内容を分ける。

4. **送信ロジック実装（Day 5-8）**
   - `services/api/app/routers/reservations.py` で予約作成処理に `BackgroundTasks` を組み込む。
   - メール送信ユーティリティ（例: `services/api/app/services/notifications.py`）を新設し、SMTP クライアントをラップ。
   - 送信失敗時のロギングと簡易リトライ（1 回）を実装。将来の Sentry 連携を考慮し例外にメタデータを付与。

5. **追加チャネル PoC（Day 8-10）**
   - LINE Notify / Slack Webhook で同じ通知テンプレートを JSON 化して送信するミニサービスを実装。`.env` でチャネル ON/OFF 切替できるようにする。

6. **テスト & QA（Day 9-12）**
   - pytest で予約生成 → 通知呼び出しをモックして検証。
   - ステージング環境で実際にメール/LINE 送信を流し、テンプレート崩れがないか確認。
   - 負荷テストとして 30 件連続送信でタイムアウトや連続失敗が起きないか計測。

7. **運用準備 & ローンチ（Day 12-14）**
   - 通知先メールアドレスの初期登録、運用マニュアル更新。
   - CI に通知設定の `.env` チェック（必須変数不足の検知）を追加。
   - リリース後 1 週間の監視項目（送信成功率、失敗アラート）を決めて運用チームに共有。

## 実行メモ

- P0 タスク（BL-001, BL-002）は直近スプリントで着手し、通知→セルフ管理の順に進める想定。
- P1 以降は P0 の進捗を見ながらロードマップ化。位置情報検索はバックエンド改修が比較的軽いため前倒し候補。
- バックログは定期的に見直し、本ドキュメントへ追記する。

### Phase 1 ステップ1: 要件整理 & チャネル決定（進行メモ）
- 送信対象候補
  - 店舗担当者: shop レコードに `notification_email`, `notification_line_token` を追加予定。
  - 運営チーム: `.env` に `ADMIN_NOTIFICATION_EMAILS` をカンマ区切りで管理。
- 通知チャネル候補
  - **メール (SMTP/SendGrid)**: MVP では SendGrid が第一候補。API キーで認証し、後に専門テンプレート機能を利用可。
  - **LINE Notify**: 店舗側が自らトークンを取得しダッシュボード設定できる想定。1 通知あたり 1000 文字まで、使い勝手は良い。
  - **Slack Webhook**: 運営内部通知用。JSON 投稿で整形が容易。
- 判明した検討事項
  - SendGrid 無しの場合でも SMTP (Gmail App Password) でバックアップを用意するか。
  - 個人情報を含む本文は暗号化 or 最小限に留める必要あり。店舗側への詳細と運営側向け詳細を分ける案。
  - 予約一件につき複数店舗担当宛（複数メール）に送る場合のスロットリング。
- 次アクション
  - SendGrid API キーの契約要否を PM と確認し、利用方針を決める（2025/09/27）。
  - `shops` テーブルへの通知先カラム追加を設計（文字数・NULL 許容）。
  - `.env` テンプレートに通知関連キーを追記する案を起票。
